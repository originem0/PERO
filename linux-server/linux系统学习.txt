  Kernel 对每个硬件设备写了一个驱动程序，这个驱动负责把"读写文件"这个动作翻译成具体的硬件操作。

  - 你对 /dev/sda（硬盘）执行"读" → 驱动把它翻译成"从磁盘某个扇区读数据"

  你看到的是文件，背后是驱动在做翻译。 你不需要知道硬盘怎么转、打印机怎么喷墨，你只管读和写。

● 📍 Linux 服务器基础 > 进程管理 | 阶段：Phase 2 Encoding | 主线进度：4/8

  三个问题，一个一个来。

  第一个："为什么自己改自己的工作目录不会导致退出？"

  你搞混了两件事。问题不是"改目录会不会导致退出"，而是"谁改的"。

  Shell 是一个持续运行的进程——你打开终端它就在，等你输入命令。cd 是 Shell 内置的，Shell 自己执行 cd，改的是自己的工作目录，改完继续等你下一条命令。

  如果 cd 是独立程序：Shell 启动一个新进程来运行 cd，这个新进程改了自己的目录，然后这个新进程执行完就退出了。Shell 的目录没变。

  关键不是"退出"，而是"改的是谁的目录"。

  第二个："Linux 和 Windows 的进程管理是不是一样的？"

  基本原理一样——都是 Kernel 管理进程，每个进程有独立空间。具体实现和工具不同，但你理解了原理，两边都能用。

  第三个："Java 里的进程和线程，跟这里的进程是同一种吗？"

  进程是同一种东西——操作系统层面的概念，不管你用什么语言写的程序，运行起来都是进程。

  线程是另一个概念：一个进程里面可以有多个线程。 线程共享同一个进程的内存空间，但各自独立执行任务。

  你的直觉很准。服务就是在后台持续运行的进程。

  进程是更大的概念——你跑个 ls，它也是进程，但跑完就退出了。服务是一种特殊的进程：启动后不退出，一直在后台等着干活。

  你的 nginx 就是服务——它启动后不退出，一直在 80 端口等着接请求。sshd 也是——一直等着你来远程登录。

  那问题来了：这些服务谁来管？ 想象一下：

  - 服务器开机，谁负责把 nginx、sshd、gunicorn 一个个启动起来？
  - nginx 突然崩了，谁发现并重启它？
  - 你想暂时关掉某个服务，跟谁说？

  答案就是 systemd——Linux 的服务管家。 它是 PID 1，系统启动后第一个跑起来的进程，然后由它按配置依次启动其他服务。

  类似 Windows 里你按 Win+R 输入 services.msc 打开的"服务管理器"——但 Linux 上没有图形界面，你通过命令行跟 systemd 交互。

  命令是 systemctl：

  systemctl start nginx     ← 启动
  systemctl stop nginx      ← 停止
  systemctl restart nginx   ← 重启
  systemctl status nginx    ← 查看状态
  systemctl enable nginx    ← 开机自启动
  systemctl disable nginx   ← 取消开机自启动

  你注意到 start 和 enable 的区别了吗？你觉得这两个为什么要分开？

  好问题。不是任何程序都能用 systemd 管。

  systemd 管理的是注册了的服务。注册的方式是在 /lib/systemd/system/ 下面有一个 .service 配置文件。你刚才看到的：

  Loaded: loaded (/lib/systemd/system/nginx.service; enabled; ...)

  这个 nginx.service 就是 nginx 的注册文件。里面写了：程序在哪、怎么启动、怎么停止、崩了要不要自动重启。

  用 apt 安装的软件，如果它是服务类型的（nginx、mysql、sshd 这种），安装时会自动帮你创建 .service 文件。 所以你装完就能 systemctl start 直接用。

  但如果你自己写了一个 Python 脚本想让它当服务跑，你得自己写一个 .service 文件注册到 systemd 里。 不注册，systemd 不认识它。

  所以辨别标准很简单：有没有 .service 配置文件。 没有就不是 systemd 管的服务。

       33 +## 1. Shell 与命令行基础

       36 +

       38 +

       40 +- **Shell（外壳）**：包在 Kernel 外面的交互层，翻译人类的命令给系统执行
       41 +
       42 +Shell 本身也是一个程序，权限取决于当前登录的用户。Linux 服务器上通常没有图形界面，默认的命令行 Shell 是 **Bash**。
       43 +
       44 +> **Windows 对比**：cmd / PowerShell 就是 Windows 的命令行 Shell。
       45 +
       46 +### Shell 的工作流
       47 +
       48 +```
       49 +你输入命令 → Shell 接收文本 → 解析命令 → 通过 PATH 找到程序 → 启动程序 → 程序与内核交互 → 返回结果

      125 +## 2. 文件系统与目录结构
      126 +
      127 +### 核心原理
      128 +
      129 +Linux 只有一个起点：**`/`（根目录）**。所有东西从这里往下长，像一棵倒着的树。
      130 +
      131 +> **Windows 对比**：Windows 有多个入口（C:、D:、E:），每个盘符一棵树。Linux 只有一棵树。
      132 +
      133 +### 挂载——路径与设备解耦
      134 +
      135 +Linux 没有盘符。第二块硬盘、U 盘等存储设备通过**挂载（mount）**接入目录树的某个位置。
      136 +
      137 +```
      138 +/            ← 根目录（第一块硬盘）
      139 +├── home/    ← 用户目录
      140 +├── data/    ← 第二块硬盘挂载在这里
      141 +└── ...
      142 +```
      143 +
      144 +**核心好处：程序只认路径，不认设备。** 硬盘换了、扩容了，只要挂载到同一个路径，上层程序完全无感。
      145 +
      146 +> **Windows 对比**：`D:\data` 这个路径本身包含了"在 D 盘上"的物理信息。换盘所有引用都要改。
      147 +
      148 +### "一切皆文件"的三层含义
      149 +
      150 +1. **普通文件和目录是文件** — 没争议
      151 +2. **硬件设备也是文件** — 硬盘是 `/dev/sda`，键盘、网卡都有对应的文件
      152 +3. **配置也是文件** — 放在 `/etc` 下的纯文本文件，不需要注册表
      153 +
      154 +**硬件怎么变成文件？** Kernel 的**驱动程序**负责翻译：你对 `/dev/sda` 执行"读"→ 驱动翻译成"从磁盘某个扇区读数据"。你看到的是文件，
          +背后是驱动在做翻译。
      155 +
      156 +**注意区分两个概念：**
      157 +- **设备文件**（如 `/dev/sda`）：硬件的文件化抽象，让你能用读写操作控制硬件
      158 +- **挂载**：把存储设备的内容接入目录树某个位置。不是所有设备文件都涉及挂载（如键盘）
      159 +
      160 +### 目录按类型组织
      161 +
      162 +Linux 不像 Windows 按软件分（Chrome 的程序、配置、日志都在 Chrome 文件夹下）。Linux 按类型统一归类：
      163 +
      164 +| 目录 | 存放内容 | 例子 |
      165 +|------|----------|------|

      200 +## 3. 用户与权限
      201 +
      202 +### 核心原理
      203 +
      204 +Linux 权限回答一个问题：**谁能对哪个文件做什么事？**
      205 +
      206 +### 为什么需要多用户
      207 +
      208 +不只是人需要不同账户，**程序也需要**。一台服务器上跑着网站、数据库、日志收集程序：
      209 +
      210 +- 如果都用 root 运行：网站被黑 → 攻击者拿到 root 权限 → 能读数据库、删日志、装后门、控制整台机器
      211 +- 如果各用专门用户运行：网站被黑 → 攻击者只拿到 `www` 权限 → 碰不到数据库和日志
      212 +
      213 +这就是**最小权限原则**：给每个用户和服务刚好够用的权限，不多给。
      214 +
      215 +### 用户分类
      216 +
      217 +- **root**：超级管理员，不受权限限制，可以做任何事。家目录在 `/root`
      218 +- **普通用户**：只能做被允许的事。家目录在 `/home/用户名`
      219 +
      220 +### 权限模型
      221 +
      222 +每个文件身上带着这些信息：
      223 +
      224 +```
      225 +-rwxr-xr-- 1 www webteam 1024 Feb 23 config.txt
      226 +↑↑↑↑↑↑↑↑↑↑   ↑    ↑
      227 +│├──┤├──┤├──┤ owner group
      228 +│ ↑   ↑   ↑
      229 +│owner group others
      230 +│
      231 +文件类型（- 普通文件，d 目录）
      232 +```
      233 +
      234 +**三类人 × 三种操作：**
      235 +
      236 +| 操作 | 字母 | 含义 |
      237 +|------|------|------|
      238 +| 读 | r | 能不能看内容 |
      239 +| 写 | w | 能不能改内容 |
      240 +| 执行 | x | 能不能当程序运行 |
      241 +
      242 +| 角色 | 含义 |
      243 +|------|------|
      244 +| owner | 文件所有者（谁创建的文件谁就是 owner） |
      245 +| group | 文件所属的用户组 |
      246 +| others | 除了 owner 和 group 之外的所有人 |
      247 +
      248 +**匹配逻辑**：系统拿你的身份去匹配——你是 owner？用第一组权限。不是 owner 但在 group 里？用第二组。都不是？用第三组（others）。
      249 +
      250 +**关键洞察：权限是绑在文件上的，不是绑在用户上的。** 同一个用户 tom，对文件 A 可能是 owner，对文件 B 可能是 others。
      251 +
      252 +### 用户组的作用
      253 +
      254 +用户组让你批量管理权限。把 tom、jerry 都加入 `webteam` 组，文件标记所属组为 `webteam`，他们就共享第二组权限。不需要一个个设。
      255 +
      256 +注意：一个文件只能标记一个 group。
      257 +
      258 +### `ls -l` 输出解读
      259 +
      260 +```
      261 +drwxr-xr-x 4 root root  4096 Feb 18 10:12 ssh
      262 +│            │  ↑    ↑     ↑       ↑         ↑
      263 +│            │ owner group 大小    时间      文件名
      264 +│            硬链接数
      265 +文件类型+权限
      266 +```
      267 +
      268 +### chmod——修改权限
      269 +
      270 +```bash
      271 +chmod u+x file      # 给 owner 加执行权限
      272 +chmod g-w file      # 去掉 group 的写权限
      273 +chmod o+r file      # 给 others 加读权限
      274 +chmod u+rwx file    # 给 owner 加全部权限
      275 +```
      276 +
      277 +- `u` = owner，`g` = group，`o` = others
      278 +- `+` 加权限，`-` 去权限
      279 +- 只有文件的 owner 和 root 才能 chmod
      280 +
      281 +> **Windows 对比**：Windows 也有文件权限，但大部分用户从不接触。Linux 上权限是日常操作。

      285 +## 4. 进程管理
      286 +
      287 +### 核心原理
      288 +
      289 +- **程序**：硬盘上的文件，静态的，死的（菜谱）
      290 +- **进程**：程序被加载到内存运行起来的实例，活的（正在炒的菜）
      291 +
      292 +同一个程序可以启动多个进程（开三个 Chrome 窗口 = 硬盘一个 chrome.exe，内存多个 Chrome 进程）。
      293 +
      294 +### 进程隔离
      295 +
      296 +**每个进程有独立的内存空间、工作目录、环境变量。Kernel 强制隔离，进程之间不能互相访问。**
      297 +
      298 +隔离的原因：
      299 +- **稳定性**：一个进程崩了不会把别的进程的内存搞乱
      300 +- **安全性**：进程之间看不到彼此的数据（网站进程看不到数据库进程的内存）
      301 +
      302 +进程之间需要通信时，必须通过 Kernel 提供的专门机制，不能直接翻别人的内存。
      303 +
      304 +### 进程 vs 线程
      305 +
      306 +- **进程**：独立的内存空间，互相隔离。类比独立的办公室。
      307 +- **线程**：一个进程内部的多个执行单元，共享内存。类比办公室里的员工共用桌子。
      308 +
      309 +在服务器上管理的主要对象是进程，不是线程。
      310 +
      311 +### Shell 与进程的关系
      312 +
      313 +```
      314 +Shell（长期运行的进程，一直等你输入）
      315 +  ├── 你输入 ls → Shell 启动新进程运行 ls → ls 执行完退出
      316 +  ├── 你输入 grep → Shell 启动新进程运行 grep → grep 执行完退出
      317 +  └── 你输入 cd → Shell 自己内部执行（不启动新进程）
      318 +```
      319 +
      320 +### PID（进程号）
      321 +
      322 +每个进程有唯一的 PID，按启动顺序递增。**PID 1 永远是 `systemd`**——系统启动后的第一个进程，所有其他进程的祖先。
      323 +
      324 +### 常用命令
      325 +
      326 +```bash
      327 +ps aux              # 查看所有进程快照（拍一张照片）
      328 +top                 # 实时动态显示（像 Windows 任务管理器）
      329 +kill 1234           # 杀掉 PID 为 1234 的进程
      330 +kill -9 1234        # 强制杀掉（进程不响应普通 kill 时用）
      331 +```
      332 +
      333 +> **Windows 对比**：`ps` / `top` = 任务管理器，`kill` = 右键"结束任务"。

      381 +## 6. 网络基础
      382 +
      383 +### 核心原理
      384 +
      385 +- **IP 地址**：找到哪台机器（大楼地址）
      386 +- **端口**：找到这台机器上的哪个程序（门牌号）
      387 +
      388 +任何需要接收网络请求的程序都要监听某个 IP + 端口。一个端口同一时间只能被一个程序占用。
      389 +
      390 +### 常见默认端口
      391 +
      392 +| 端口 | 服务 |
      393 +|------|------|
      394 +| 22 | SSH（远程登录） |
      395 +| 80 | HTTP（网站） |
      396 +| 443 | HTTPS（加密网站） |
      397 +| 3306 | MySQL |
      398 +
      399 +浏览器访问 `http://xxx` 时自动访问 80 端口，不需要写 `:80`。
      400 +
      401 +### 网络接口与监听地址
      402 +
      403 +一台机器可以有多个网络接口，每个有自己的 IP：
      404 +
      405 +```
      406 +lo（回环）:     127.0.0.1         ← 本机跟自己通信，断网也能用
      407 +eth0（网卡）:   36.151.146.104    ← 公网 IP
      408 +```
      409 +
      410 +程序监听时选择：
      411 +
      412 +| 监听地址 | 含义 | 谁能访问 |
      413 +|---------|------|---------|
      414 +| `127.0.0.1:8000` | 只监听本机回环 | 只有服务器本机 |
      415 +| `36.151.146.104:80` | 只监听公网网卡 | 外部通过公网 IP |
      416 +| `0.0.0.0:80` | 监听所有接口（通配符） | 本机 + 外部都能访问 |
      417 +
      418 +`0.0.0.0` 不是真实 IP，是"我全都要"的声明。
      419 +
      420 +### 反向代理架构
      421 +
      422 +典型的服务器架构：
      423 +
      424 +```
      425 +外部用户 → nginx（0.0.0.0:80）→ gunicorn（127.0.0.1:8000）→ 你的代码
      426 +```
      427 +
      428 +为什么不让应用服务器直接对外？
      429 +- **安全**：nginx 过滤恶意请求，应用服务器不暴露在公网
      430 +- **统一接待**：nginx 处理 HTTPS、负载均衡、静态文件
      431 +- **最小暴露**：gunicorn 只监听 `127.0.0.1`，外面碰不到
      432 +

      434 +
      435 +```
      436 +程序在跑吗？           → ps aux | grep nginx
      437 +端口在监听吗？         → ss -tlnp
      438 +本机能访问自己吗？     → curl localhost:80
      439 +网络通吗？             → ping 对方IP

      441 +```

      443 +**排查原则：从里往外查。** 先确认程序没问题，再看网络。大部分时候问题在前两步。

      535 +## 8. 基本安全实践
      536 +
      537 +### 核心原则
      538 +
      539 +安全的核心就是最小权限：**能不暴露的就不暴露。**
      540 +
      541 +### 默认状态有多危险
      542 +
      543 +你的服务器默认：SSH 监听 `0.0.0.0:22`，用 root + 密码登录。
      544 +
      545 +攻击者需要猜的信息：
      546 +- IP — 扫描 IP 段就能找到
      547 +- 端口 — 22，人人都知道
      548 +- 用户名 — root，所有 Linux 都有
      549 +- **密码 — 唯一需要猜的**
      550 +
      551 +自动化脚本 24 小时不停试，一秒几百个密码。你的服务器现在大概率已经被人扫了。
      552 +
      553 +验证：
      554 +```bash
      555 +grep "Failed password" /var/log/auth.log | tail -20
      556 +```
      557 +
      558 +### SSH 加固三件套
      559 +
      560 +**1. 密钥登录，禁用密码**
      561 +
      562 +密钥原理：你的电脑生成一对钥匙（公钥 + 私钥）。公钥放服务器，私钥留本地。登录时做数学验证，私钥不在网络上传输，不存在被暴力猜解的
          +可能。
      563 +
      564 +```bash
      565 +# 本地电脑生成密钥对
      566 +ssh-keygen -t ed25519
      567 +
      568 +# 把公钥传到服务器
      569 +ssh-copy-id user@your-server-ip
      570 +
      571 +# 确认密钥登录成功后，禁用密码登录

❯ id -u loanapp &>/dev/null || useradd -m -s /bin/bash loanapp 解释一下这条命令是什么意思

● 📍 Linux 服务器基础 > 实战解读 | 补充

  正好用我们学过的知识来拆：

  id -u loanapp &>/dev/null || useradd -m -s /bin/bash loanapp

  分三部分：

  第一部分：id -u loanapp
  - id 是一个程序，查看用户信息
  - -u 选项：只输出用户的 UID（用户 ID 号）
  - loanapp 参数：查的是这个用户
  - 如果 loanapp 用户存在 → 命令成功（返回 0）
  - 如果不存在 → 命令失败（返回非 0）

  第二部分：&>/dev/null
  - &> 是重定向，把标准输出和错误输出都扔到 /dev/null
  - /dev/null 是一个特殊的设备文件——黑洞，写进去的东西全部消失。目的是不让输出显示在屏幕上，只关心命令成功还是失败

  第三部分：||
  - 这是逻辑"或"。意思是：左边失败了，才执行右边
  - 左边成功了（用户已存在）→ 右边不执行
  - 左边失败了（用户不存在）→ 执行右边

  第四部分：useradd -m -s /bin/bash loanapp
  - useradd 创建新用户
  - -m 自动创建家目录（/home/loanapp）
  - -s /bin/bash 设置这个用户的默认 Shell 为 Bash
  - loanapp 新用户的名字

  整句话翻译成人话：如果 loanapp 用户不存在，就创建它；已经存在就什么都不做。

  这是一种幂等写法——跑一次和跑十次效果一样，常用于自动化脚本里，避免重复创建用户时报错。

● 好，三个问题加逐行解读。

  1. 为什么不打包成 .deb？

  你自己的项目，用户就你一个人，服务器可能就一两台。打 .deb 包需要按特定格式组织文件结构、写控制文件、处理依赖声明——工程量不小，收益几乎为零。

  .deb 包适合：分发给成千上万用户安装的通用软件（nginx、mysql 这种）。

  你自己的项目，一个 Shell 脚本跑一遍就部署好了，这是最实际的做法。

  2. .sh 是什么？

  .sh 是 Shell 脚本文件。就是把你在命令行里一条条输入的命令，写到一个文件里，一次性执行。

  Windows 对比：.bat 批处理文件。你在 cmd 里手动输的命令写成 .bat，双击就跑。.sh 是 Linux 版的 .bat。

  第一行 #!/bin/bash 告诉系统：用 Bash 来执行这个脚本。

  set -e 的意思是：任何一条命令失败就立刻停止，不继续往下跑。防止前面出错了后面还在瞎跑。

  3. 这个脚本跟你今天学的知识哪些有关？

  几乎全有关。逐段对应：

  模块 5（包管理）：
  apt update && apt upgrade -y
  apt install python3 python3-venv python3-pip nginx sqlite3 -y
  用 apt 安装依赖。-y 是选项，意思是"自动确认，别问我"。&& 是"前面成功了才执行后面"（跟 || 相反）。

  模块 3（用户与权限）：
  id -u loanapp &>/dev/null || useradd -m -s /bin/bash loanapp
  创建专门的应用用户。为什么不直接用 root 跑项目？最小权限原则——项目被攻破时攻击者只有 loanapp 的权限。

  chown -R loanapp:loanapp /opt/loanmanager
  chown = change owner。把整个目录的 owner 和 group 都改成 loanapp。-R 是递归，目录下所有文件都改。

  chmod 600 /opt/loanmanager/.env
  .env 里存着 SECRET_KEY（密钥），600 是权限的数字写法：owner 可读可写（rw-），group 和 others 什么都不能干（--- ---）。只有 loanapp
  能读这个密钥，别的用户看不到。

  补充：chmod 600 是数字写法，r=4, w=2, x=1，加起来就是权限值。6=4+2=rw，0=什么都没有。所以 600 = rw-------。

  模块 2（文件系统）：
  mkdir -p /opt/loanmanager/logs
  mkdir -p /opt/loanmanager/backups
  mkdir -p /opt/loanmanager/instance
  创建项目目录结构。-p 表示父目录不存在就自动创建。项目放在 /opt——这个目录通常用来放第三方/可选的软件。

  模块 4（进程）：
  sudo -u loanapp bash << 'DEPLOY'
  ...
  DEPLOY
  sudo -u loanapp 意思是"以 loanapp 用户的身份执行"。这样创建的虚拟环境和安装的包都属于 loanapp，不属于 root。又是最小权限原则。

  << 'DEPLOY' 到 DEPLOY 之间的内容是一段嵌入的脚本文本，叫 heredoc——把多行内容当作输入传给 bash。

  模块 1（Shell 基础）：
  整个文件就是管道、重定向、选项、参数的综合运用。echo 输出提示信息、&>/dev/null 重定向到黑洞、|| 和 && 控制执行逻辑。

  总结：这个脚本做了一件事——在一台全新服务器上，用一条命令完成所有部署准备。 这就是我们在模块 1
  学的命令行第二个优势：命令是文本，可以被保存、复制、自动化执行。

● 两个问题。

  为什么 IP 不对？

  198.18.0.x 不是你服务器的公网 IP，而是你电脑上那个 Meta 适配器的地址段。还记得之前你的 ipconfig 结果吗？

  未知适配器 Meta:
     IPv4 地址: 198.18.0.1

  这说明你本地有一个代理或 VPN 工具（Clash、Surge 之类的）拦截了 DNS 解析，把域名解析到了本地代理地址。请求实际是：你 → 本地代理（198.18.0.x）→
  代理转发到你的真实服务器（36.151.146.104）。

  所以不是域名解析错了，是你的代理在中间做了一层拦截。你关掉代理再 ping，应该能看到真实的公网 IP。

  加 www 和不加的区别

  sharonzhou.site 和 www.sharonzhou.site 是两条不同的 DNS 记录。

  - sharonzhou.site → 解析到 198.18.0.87（裸域名）
  - www.sharonzhou.site → 解析到 198.18.0.88（www 子域名）

  IP 不一样说明你在 DNS 里配了两条记录。它们可以指向同一台服务器（一般都这么做），但在 DNS 层面是两个独立的名字。

  类比：sharonzhou.site 是大楼名字，www.sharonzhou.site 是大楼里某个部门的名字。它们可以在同一栋楼里，但门牌号可以不同。

  在你的代理环境下两个被分配了不同的本地地址（87 和 88），但最终可能都指向同一台服务器。你可以在服务器上用 nginx -T 看看 nginx
  配置里是不是同时处理了这两个域名。
